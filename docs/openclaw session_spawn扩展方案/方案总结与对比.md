## 各方案一览

| 方案 | 名称 | taskId 传入方式 | LLM 感知 | 实现复杂度 | 可靠性 | 是否修改源码 |
|------|------|---------------|---------|-----------|-------|------------|
| **1A** | label 编码 + after_tool_call | label 字段 | ❌ | ⭐ | ⭐⭐⭐⭐ | 否 |

## 关键约束（已源码验证）

### 约束 1：不能同名替换核心工具

`src/plugins/tools.ts:112`：插件工具名与核心工具名冲突时直接报错跳过。

```typescript
if (nameSet.has(tool.name) || existing.has(tool.name)) {
  log.error(`plugin tool name conflict: ${tool.name}`);
  continue; // ← 跳过该工具
}
```

❌ **无法注册名为 `sessions_spawn` 的插件工具。**

### 约束 2：before/after hook 均无 toolCallId

`src/plugins/types.ts:479-497`：两个 hook 的 event 结构中都没有 `toolCallId`。

```typescript
type PluginHookBeforeToolCallEvent = { toolName: string; params: Record<string, unknown> };
type PluginHookAfterToolCallEvent  = { toolName: string; params: Record<string, unknown>;
                                       result?: unknown; error?: string; durationMs?: number };
```

⚠️ **方案 2 只能用 `params` 指纹（task + label）作为跨 Hook 状态的 key。**

### 约束 3：before_tool_call 可以修改参数

```typescript
type PluginHookBeforeToolCallResult = {
  params?: Record<string, unknown>; // ← 可修改参数
  block?: boolean;
  blockReason?: string;
};
```

✅ **before_tool_call 可以剥离 `clawteam_taskId` 避免传入原始工具造成错误。**

## 推荐方案

### 场景 A：快速落地，低成本接入
> 推荐 **方案 1A（label 编码）**

适合已有 sessions_spawn 调用，只需追加一个固定格式的 label 前缀。无需改动任何工具或 schema，改动量最小。

**调用方式：**
```typescript
sessions_spawn({
  task: "Analyze security",
  label: "[TASK:CLAWTEAM-12345] Security Analysis",
})
```

### 场景 B：需要前置验证 taskId 合法性
> 推荐 **方案 2（双 Hook）**

在 before_tool_call 中调用外部接口验证 taskId 是否存在，验证通过后暂存；after_tool_call 中完成追踪。

**调用方式：** 与方案 1A 相同，labelq 编码传入。

### 场景 C：长期维护，需要 LLM 直接感知 taskId 参数
> 推荐 **方案 3（注册新工具）**

注册 `sessions_spawn_tracked` 工具，将 `clawteam_taskId` 作为独立的 schema 字段暴露给 LLM。适合产品化、长期维护的场景。

**调用方式：**
```typescript
sessions_spawn_tracked({
  clawteam_taskId: "CLAWTEAM-12345",
  task: "Analyze security",
  label: "Security Analysis",
})
```

## 共用的 tracker.ts

无论选择哪个方案，`track_session` 函数实现相同：

```typescript
// tracker.ts
export interface TrackSessionOptions {
  apiEndpoint?: string;
  apiKey?: string;
  logger?: { info: Function; warn: Function; error: Function; debug: Function };
}

export async function trackSession(
  clawteam_taskId: string,
  childSessionKey: string,
  options: TrackSessionOptions = {},
): Promise<void> {
  const { apiEndpoint, apiKey, logger } = options;

  logger?.info(`[ClawTeam] track_session("${clawteam_taskId}", "${childSessionKey}")`);

  // 本地写日志
  await appendLocalLog({ clawteam_taskId, childSessionKey });

  // 若配置了远端接口，发送追踪数据
  if (apiEndpoint) {
    const response = await fetch(apiEndpoint, {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        ...(apiKey ? { Authorization: `Bearer ${apiKey}` } : {}),
      },
      body: JSON.stringify({
        taskId: clawteam_taskId,
        sessionKey: childSessionKey,
        trackedAt: new Date().toISOString(),
      }),
    });
    if (!response.ok) {
      throw new Error(`Track API returned ${response.status}`);
    }
  }
}

async function appendLocalLog(entry: object): Promise<void> {
  const fs = await import("fs/promises");
  const path = await import("path");
  const os = await import("os");
  const dir = path.join(os.homedir(), ".openclaw", "clawteam");
  await fs.mkdir(dir, { recursive: true });
  await fs.appendFile(
    path.join(dir, "sessions.jsonl"),
    JSON.stringify({ ...entry, ts: new Date().toISOString() }) + "\n",
    "utf-8",
  );
}
```

## 分阶段演进路径

```
第 1 阶段（快速验证）
  → 方案 1A（label 编码 + after_tool_call hook）
  → 验证 track_session 能正确关联 taskId 和 childSessionKey

第 2 阶段（增加校验）
  → 升级为方案 2（双 Hook + 前置验证）
  → 确保 taskId 合法性，增加错误日志

第 3 阶段（产品化）
  → 升级为方案 3（新工具 sessions_spawn_tracked）
  → clawteam_taskId 成为独立字段，LLM 自然感知
  → System Prompt 引导 LLM 在有 taskId 时自动选用新工具
```
